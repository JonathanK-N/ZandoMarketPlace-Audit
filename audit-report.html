01 · Structure du code

🏗️ Architecture des applications Django

ZandoMarketPlace est découpé en 5 apps Django. Les flèches montrent les dépendances réelles trouvées dans le code (imports, ForeignKey). Plus une app a de connexions entrantes, plus elle est centrale dans le système.

👤

accounts

Utilisateur custom (client/seller/admin) + 3 profils spécialisés. Modèles complets, 0 vue active.

🛍️

products

Catalogue, variantes, rabais, avis, wishlist. App la plus aboutie : vues, templates, admin riche.

🛒

orders

Panier et commande modélisés en base, aucune vue, aucune route active.

💳

payments

Modèle Payment seul. Stripe en dépendance, 0 ligne d'intégration.

📊

dashboard

Censée gérer l'administration globale. Entièrement vide — code généré par défaut uniquement.

02 · Base de données

🧬 Schéma relationnel — 3D animé & MCD formel

Deux lectures complémentaires du même schéma réel : une vue 3D immersive pour explorer, et un Modèle Conceptuel de Données classique (notation Mocodo) pour les cardinalités précises.

🧬 Visualisation 3D interactive disponible dans la version web (fichier schema-3d.html) — non reproductible sur papier/PDF.
🖱️ Interactif en direct ci-dessus — fonctionne aussi en plein écran via schema-3d.html
📂 Source éditable : docs/mcd/zandomarketplace.mcd — régénérable avec python -m mocodo --input zandomarketplace.mcd. Notation classique : entités à gauche (attribut souligné = identifiant), associations en losange, cardinalités (0,1 / 1,1 / 0,N / 1,N) sur chaque branche.
03 · Audit technique approfondi

🔍 Toutes les failles, classées par domaine

Revue exhaustive ligne par ligne du code reçu, complétée par des tests fonctionnels réels sur le déploiement Railway. Chaque finding indique le fichier, la ligne, le code fautif et l'impact concret.

ZMP-2026-001Critique

Aucun dépôt Git — projet jamais versionné

Racine du projet

Le code n'existait sous aucune forme de contrôle de version. Zéro historique, zéro sauvegarde, zéro rollback possible en cas d'erreur humaine.

Impact : perte totale du travail possible en un clic (suppression accidentelle, crash disque).
ZMP-2026-002Critique

Secrets en clair, sans protection

.env
SECRET_KEY=django-insecure-7$y9x!kq8p@r#9*abc1234567890abcdef
DEBUG=True

La clé qui signe les cookies de session et les tokens CSRF était stockée en clair, sans .gitignore pour empêcher un commit accidentel. DEBUG=True expose les stack traces complètes (chemins serveur, requêtes SQL) à tout visiteur en cas d'erreur.

Impact : usurpation de session, falsification CSRF, fuite d'informations serveur.
ZMP-2026-004Majeur

ALLOWED_HOSTS ouvert à tous

zandomarket/settings.py
ALLOWED_HOSTS = ['*']

Accepte n'importe quel en-tête Host — vecteur classique d'attaque par Host Header Injection (empoisonnement de cache, liens de réinitialisation de mot de passe forgés).

ZMP-2026-005Majeur

CORS configuré... mais jamais actif

settings.py — INSTALLED_APPS vs MIDDLEWARE
INSTALLED_APPS = [..., 'corsheaders', ...]  # présent

MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    # 'corsheaders.middleware.CorsMiddleware',  ← JAMAIS AJOUTÉ
    'django.contrib.sessions.middleware.SessionMiddleware',
    ...
]

corsheaders est listé comme app installée, donnant l'illusion d'une protection CORS. Le middleware réel n'a jamais été branché : la protection n'a jamais été active, à aucun moment de la vie du projet.

ZMP-2026-017Mineur

Ressource CDN sans Subresource Integrity

base.html — ligne 15
<link href="https://cdnjs.cloudflare.com/.../font-awesome/6.4.2/css/all.min.css" rel="stylesheet">
<!-- pas d'attribut integrity ni crossorigin -->

Si ce CDN est un jour compromis, du CSS/JS arbitraire pourrait être injecté sans aucune alerte du navigateur.

ZMP-2026-024Majeur

order_by('?') — tri aléatoire au niveau SQL sur chaque vue de la page d'accueil

apps/products/views.py — ligne 28
recommended_products = Product.objects.filter(
    is_active=True, is_approved=True
).order_by('?')[:12]   # ⚠️ ORDER BY RANDOM() en PostgreSQL

Sous PostgreSQL/MySQL, order_by('?') force la base à trier la table entière en mémoire avec une valeur aléatoire par ligne avant de garder 12 résultats. Sur une table de quelques milliers de produits, c'est déjà coûteux ; sur une table de plusieurs centaines de milliers de lignes, c'est une requête qui peut prendre plusieurs secondes — exécutée à chaque chargement de la page d'accueil, par chaque visiteur.

Impact : la page d'accueil devient de plus en plus lente à mesure que le catalogue grossit — la pire des relations entre "succès commercial" et "performance".
ZMP-2026-010Majeur

Problème N+1 — jusqu'à 40 requêtes SQL juste pour afficher des étoiles

apps/products/models.py — Product.average_rating / rating_count
@property
def average_rating(self):
    return self.reviews.aggregate(avg=models.Avg('rating'))['avg'] or 0  # 1 requête

@property
def rating_count(self):
    return self.reviews.count()                                          # 1 requête

Appelées dans le template pour chacun des 12 + 8 = 20 produits affichés en page d'accueil. .aggregate() et .count() ignorent totalement le cache de prefetch_related() et retournent systématiquement en base — même là où le prefetch est pourtant déjà présent dans la vue.

Impact mesuré : jusqu'à 40 requêtes SQL additionnelles par chargement de la page d'accueil, uniquement pour le rendu des étoiles.
ZMP-2026-025Majeur

N+1 sur les publicités sponsorisées

views.py ligne 21 + home.html ligne 93
# views.py — pas de select_related('product')
ads = Ad.objects.filter(is_active=True).order_by('-priority')[:10]
<!-- home.html — déclenche 1 requête PAR publicité -->
{{ ad.product.final_price }} {{ ad.product.currency }}

Sans select_related('product'), chaque accès à ad.product dans la boucle déclenche sa propre requête — jusqu'à 10 requêtes supplémentaires pour la seule section "Offres sponsorisées".

ZMP-2026-026Mineur

Pas de pooling de connexions DB

settings.py — DATABASES

CONN_MAX_AGE n'est jamais défini : Django ouvre et ferme une nouvelle connexion PostgreSQL à chaque requête HTTP au lieu de la réutiliser. Coût négligeable à faible trafic, mais devient un goulot d'étranglement réel dès que le nombre de visiteurs simultanés augmente sur Railway.

ZMP-2026-027Mineur

Aucun index sur les champs filtrés en permanence

apps/products/models.py
is_active = models.BooleanField(default=True)     # filtré dans CHAQUE requête produit
is_approved = models.BooleanField(default=False)  # filtré dans CHAQUE requête produit
is_daily = models.BooleanField(default=False)     # filtré sur la page d'accueil

Aucun de ces champs n'a db_index=True, malgré un usage systématique dans les .filter() de la page la plus visitée du site. Sans index, PostgreSQL doit parcourir séquentiellement la table entière à chaque appel.

ZMP-2026-028Mineur

Aucune pagination — plafond de scalabilité programmé

apps/products/views.py::home()

Les requêtes sont limitées par des slice codés en dur ([:12], [:8]). Il n'existe aucune page de catégorie, aucune pagination, aucun "voir plus" fonctionnel : au-delà de la page d'accueil, le catalogue est tout simplement inaccessible, quelle que soit sa taille.

ZMP-2026-029Mineur

Historique de vues sans limite ni déduplication

apps/products/views.py::product_detail() — ligne 79
if request.user.is_authenticated:
    ProductView.objects.create(user=request.user, product=product)
    # une ligne créée à CHAQUE requête, même un simple refresh (F5)

Aucune contrainte d'unicité, aucun throttling. Un utilisateur qui rafraîchit frénétiquement une fiche produit (ou un script automatisé) peut générer un nombre illimité de lignes — vecteur de saturation de table.

ZMP-2026-011Mineur

MaxValueValidator est pourtant déjà importé en haut du fichier (utilisé sur rating) mais jamais appliqué ici. Un rabais de 150% produit un final_price négatif.

ZMP-2026-012Mineur

apps/accounts/models.py
@property

user_type (texte modifiable) et is_staff/is_superuser (vraies permissions Django) ne sont jamais synchronisés — confusion garantie entre "rôle affiché" et "droits réels".

ZMP-2026-016Mineur

ZMP-2026-008bMajeur

🐛 Bug Django classique — images invisibles en production

zandomarket/urls.py (avant correction)
if settings.DEBUG:
